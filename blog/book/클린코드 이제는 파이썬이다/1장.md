# 1장 에러 메시지 파악과 고수의 도움을 받기 위한 질문 에티켓
프로그래밍 학습하는 개발자들은 컴퓨터 앞에 앉아 독학하는 경우가 많다. 내가 프로그래밍을 하며 부딪힌 문제에 대하여 답을 줄 수 있는 사수가 있거나 도움을 줄 수 있는 시니어 개발자가 없으면 주야장천 컴퓨터 앞에 앉아 인터넷 검색에만 매달리게 된다.

<br>

다행인 점은, 우리가 궁굼해 하는 대부분의 문제는 이미 다른 개발자가 인터넷에 질문을 올려놨을 것이다.프로그래머로서 스스로 답을 찾는 습관이나 알고리즘, 데이터 구조 지식을 쌓는 일도 중요하지만 훨씬 더 중요한 일은 내가 부딪힌 문제점을 어떻게 질문하냐는 것이다.

<br>

## 파이썬 에러 메시지를 읽는 법
어떤 프로그래머든 개발하며 에러 메시지를 읽는 경우가 분명히 있을 것이다. 하지만 에러 메시지가 익숙하지 않은 초보 개발자는 이러한 메시지는 외국어를 읽는 것처럼 분석을 하고 해석을 해야하는 느낌이 들 것이다. 이런 메시지에는 프로그래밍을 하면서 무엇이 잘못됐는지를 이해할 수 있는 해답이 담겨있다. 이 답을 찾는 과정은 2단계로 나뉘는데 첫 번째 추적정보(Traceback)을 조사하는 것, 두 번째 에러 메시지를 인터넷에 검색하는 것이다.

<br>

### 추적정보 조사
파이썬 프로그램은 코드가 except문으로 처리할 수 없는 예외가 발생하면 충돌(crash)을 낸다. 이렇게 충돌이 발생했을 때 위에서 이야기한 에러 메시지를 보여주는데 우리는 여기서 추적정보(스택 추적)를 읽어 에러를 유추하거나 구글링을 통해 해결방법을 찾아야한다.

<br>

추적정보를 읽는 연습을 위해, 아래와 같이 버그가 있는 프로그램을 입력해보자.
### 코드 1
```python
def a():
    print('Start of a()')
    b() # b() 함수 호출

def b():
    print('Start of b()')
    c() # c() 함수 호출

def c():
    print('Start of c()')
    42 / 0 # 이 식은 0으로 나누기 에러를 발생시킴

a() # a() 함수 호출
```
위의 코드는 a()함수가 b()를, b()함수가 c()를 호출하며 c()안에 있는 42 / 0표현식을 읽으며 에러를 발생 시킨다. 출력결과는 다음과 같다.

<br>


해당 코드를 실행시키면 위와 같은 에러 메시지가 발생하는 것을 확인 할 수 있을 것이다.

<br>

### Tracback (most recent call last)
```bash
ZeroDivisionError                         Traceback (most recent call last)
```
<span style='color:orange;'>이 메시지는 첫째 함수 호출을 시작해서 가장 최근에 호출된 함수로 끝나는 순서에 따라 추적정보가 나열되었음을 알려준다.</span>

<br>

### File ~~, line , in <module>
```bash
Cell In [1], line 11, in c()
      9 def c():
     10     print('Start of c()')
---> 11     42 / 0

ZeroDivisionError: division by zero
```
다음은 추적정보의 첫 번째 함수 호출이다. 이 내용은 <span style='color:orange;'>프레임 요약으로</span>, 프레임 객체 내부의 정보를 보여준다. 함수가 호출되면 함수 호출이 끝난 후 반환될 코드 위치와 함께 지역 변수 데이터가 프레임 객체에 저장된다. 프레임 객체는 함수가 호출될 때 생성되고 반환될 때 소멸된다. 추적정보에는 충돌로 이어진 각 프레임에 대한 프레임 요약 정보가 표시된다.

숫자를 0으로 나누는 것은 수학적으로 불가능 하고 "ZeroDivisionError: division by zero"라는 메시지를 읽는 순간 아 0으로 나눈 부분이 잘못되었구나 라는 것을 알 수 있을 것이다.

<br>

### 코드 2
```python
def spam(number1, number2):
    return number1 / (number2 - 42)

spam(101, 42)
```

<br>

### 에러 메시지 2
```bash
ZeroDivisionError                         Traceback (most recent call last)
Cell In [1], line 4
      1 def spam(number1, number2):
      2     return number1 / (number2 - 42)
----> 4 spam(101, 42)

Cell In [1], line 2, in spam(number1, number2)
      1 def spam(number1, number2):
----> 2     return number1 / (number2 - 42)

ZeroDivisionError: division by zero
```
프로그램을 실행한 출력 결과는 다음과 같다. 에러 메시지는 동일하지만 "return number1 / (number2 - 42)"라는 부분을 읽어도 문제가 확연히 드러나지 않는다. / 연산자에서 나눗셈이 이뤄지고 (number2 - 42) 표현식은 0으로 계산됐어야 한다고 추론할 수 있다. 이렇게 되면 spam()함수의 number2파라미터가 42로 설정될 때마다 문제가 발생한다는 결론을 얻는다.

<br>

### 코드 3
```python
print('Hello'
print('How are you?')
```

### 에러 메시지 3
``` bash
  Cell In [2], line 1
    print('Hello'
         ^
SyntaxError: '(' was never closed
```
때로는 추적 정보가 버그의 진짜 원인을 초래한 곳이 아니라 바로 뒤에 에러가 있다고 잘못 알려주기도 한다. 그 이유는 파이썬 인터프리터가 두 번째 행을 읽기 전까지 구문 에러를 인지하지 못했기 때문이다. 추적정보는 어디가 잘못됐는지를 나타낼 수는 있지만, 그것이 항상 버그의 실제 원인의 위치와 같지 않다는 것을 주의 하자.

프레임 요약에 버그 파악을 위한 충분한 정보가 없거나 버그의 진짜 원인이 추적정보에서 보이지 않는 이전 행에 있는 경우, 디버거를 사용해 단계별로 흐름을 따라가거나 로그를 확인해 원인을 찾아야 한다.

## 에러 메시지 검색
에러 메시지가 너무 짧아서 완전한 문장 형태를 띠지 않는 경우도 있다. 프로그래머들이 워낙 자주 접하기 때문에 에러 메시지는 온전한 설명을 한다기보다 무언가를 상기시키려는 목적이 크다. 에러 메시지를 처음 접한 경우 해당 메시지를 복사해서 인터넷 검색창에 붙여 넣으면, 그 에러의 의미와 가능성이 높은 원인이 무엇인지에 대한 자세한 설명을 얼마든지 볼 수 있다.

<span style='color:orange;'>에러 메시지 주위를 따옴표로 묶으면 정확한 구문을 찾는 데 도움이 되고 또 python이라는 단어를 추가하면 검색 범위도 좁힐 수 있다.</span>

```text
# 검색어
python "zeorDivisionError: division by zero"
```

<br>

## 린터를 활용한 에러 방지
에러 발생을 방지하기 위한 가장 좋은 방법은 실수를 하지 않는 것이다. linter라고도 부르는 lint소프트웨어는 소스 코드를 분석해 잠재적인 에러를 경고하는 응용프로그램이다.

린터가 모든 에러를 잡아내는 것은 아니지만 정적 분석(실행하지 않고 소스 코드를 검사하는 것)을 통해 오타로 인한 일반적인 에러를 알아낼 수 있다. 프로그램을 실행해서 충돌하는 상황을 보고 추적정보를 읽은 다음에 소스 코드에서 행을 찾아 오타를 고쳐야 한다.

<br>

## 프로그래밍 조언을 구하는 방법
인터넷 검색이나 린터로 문제가 해결되지 않을 때는 인터넷상에서 다른 개발자들에게 프로그래밍과 관련한 도움을 요청할 수가 있다. 도움을 요청할 때에도 지켜야할 에티켓이 존재한다. 아래에 다른 사람에게 프로그래밍 질문을 하는 경우 흔히 저지르는 실수들이 있다.

- 본론으로 돌입해 바로 질문하지 않고, 질문해도 괜찮으지를 먼저 묻는 것
- 직접 질문하지 않고 슬쩍 돌려 말하는 것
- 적절하지 않은 포럼이나 웹사이트에 질문하는 것
- "도움 구함", "문제가 있습니다." 등의 추상적인 게시물 표제나 이메일 제목을 쓰는 것
- "프로그램이 안돌아가요."라고만 말하며, 구체적으로 어떻게 작동하기 원하는지를 설명하지 않는 것
- 전체 에러 메시지를 명시하지 않는 것
- 코드를 공유하지 않는 것
- 형편없이 포매팅된 코드를 공유하는 것
- 자신이 이미 시도해본 내용을 설명하지 않고 넘어가는 것
- 운영체제나 버전 정보를 제공하니 않는 것
- 자신을 위해 누군가 대신 프로그램을 짜주기를 요청하는 것

이러한 나쁜 습관은 남들로 하여금 도와주고 싶은 생각을 달아나게 한다. 우리를 도와줄 사람들의 여러 실행에 부딛힌 문제를 재현해 볼것이다.

<br>

### 1. 미리 충분한 정보를 제공해 불필요한 정보를 제공해 불필요한 커뮤니케이션을 줄이자.
누군가에게 "질문 좀 해도 될까요?"라고 묻는 것은 상대에게 도움을 베풀어줄 수 있는지를 짧고 예의바르게 묻는 것이다. 하지만 도움을 주는 사람은 본인이 여유가 될 때 답변을 주기 때문에 질문을 해도 되지는 묻는 것 보다는 처음 들을 작성할 때 모든 정보를 담아 질문을 하는 것이 좋다.

<br>

### 2. 물음표가 붙은 질문 형식으로 문잘을 서술하자.
대부분의 경우, 문제를 설명할 때 자신이 현재 무엇에 대해 이야기하고 있는지를 사람들이 이미 알고 있다고 가정하기 쉽다. 하지만 프로그래밍은 광범위한 분야라 우리가 겪고 있는 문제의 특정 영역에 대해 남들은 아무런 경험도 없을 수 있다. 그렇기 때문에 "나는 ...을 원한다", 또는 "코드가 동작하지 않는다."로 시작하는 문장도 질문이 무엇인지 암시할 수 있지만 정확히 물음표로 끝나는 문장으로 구성해서 문장을 서술하는 것이 좋다.

<br>

### 3. 적절한 웹사이트를 찾아 질문하자.
파이썬에대한 질문을 자바스크립트 포럼에 묻거나 알고리즘 질문을 네트워크 보안 메일링 리스트에 묻는다면 답변이 돌아 올 수는 있겠지만 질문 종휴에 따라 적절한 사이트에 가서 질문을 하는 것이 좋다.

<br>

### 4. 질문을 요약한 제목을 달자.
"도와주세요", "왜 이게 안될까요?" 같은 일반적인 제목은 너무 모호하기 때문에 포럼에서 질문을 잘 검색할 수 있도록 질문에 대한 키워드를 제목에 달아 질문을 올리자.

<br>

### 5. 코드로 어떤 일을 하고 싶은지 설명하자.
도움을 주려는 입장에선 우리가 올린 질문의 의도를 정확하게 파악할 수가 없다. 그렇기 때문에 지금 작성된 코드가 어떠한 프로그램을 의도한 것인지 함께 설명 하는 것이 좋다.

<br>

### 6. 전체 에러 메시지를 제공하자.
추적정보를 포함한 전체 에러 메시지를 복사해서 붙여 넣어야 한다. "out of range에러가 발생했어요"라고 하면 에러가 어떻게 발생한 지는 유추할 수 있지만 어떤 부분에서 발생한 것인지 확인 할 수 없기 때문에 질문을 하려면 전체를 공개해서 제공하자.

<br>

### 7. 전체 코드를 공유하자.
전체 에러 메시지와 추적 정보뿐만 아니라 전체 프로그램에 대한 소스코드도 제공해야 한다. 언제나 에러를 재현할 수 있는 "최소한의 완전하고 재현 가능한" 예제를 만들어라.

<br>

### 8. 적절한 포매팅으로 코드를 읽기 쉽게 만들자.
코드를 공유하는 목적은 우리를 도와줄 사람들이 프로그램을 실행해 우리가 겪는에러를 그대로 재현하게 만드는 것이다. 따라서 복사해서 받는다거나 메일로 소스 전체를 받을때 포메팅이 되어 있지 않으면 코드를 분석하기도 쉽지않고 찾기도 힘들다.

<br>

### 9. 사람들에게 우리가 무엇을 시도했는지를 알리자.
질문을 게시할 때는 우리가 이미 시도한 내용과 그 결과들을 사람들에게 알려야 한다. 이와 같은 정보를 공유하면 그들도 잘못된 시도를 되풀이하는 노력을 줄ㅇ리 수 있으며 또 우리가 또한 이 문제를 해결하는데 노력을 기울였다는 사실을 호소할 수 있다.

<br>

### 10. 우리의 설정 사항을 공유하자.
우리의 컴퓨터의 특정 설정은 프로그램이 어떻게 실행되고 어떤 에러가 발생하는지에 영향을 미칠 수 있다. 다양한 사람들이 해당 문제를 각자 컴퓨터에서 재현해볼 수 있도록, 우리는 다음과 같은 컴퓨터 정보를 제공해야 한다.

- "윈도우 10 프로페셔널 버전"  또는 "맥OS 카탈리나" 같은 운영체제 종류와 버전
- "파이썬 3.8"또는 "파이썬 3.6.6" 같은 사용하고 있는 언어의 버전
- "장고 2.1.1"과 같이 우리가 사용하고 있는 서드파티 모듈 버전

대부분 이런 정보가 꼭 필요하진 않을 것이다 하지만 서로 간에 불필요한 질문이 오가는 것을 방지하기 위해 처음 게시할 때부터 정보를 제공하는 편이 좋다.


## 정리
  프로그래머들이 만들어낸 인터넷에는 프로그래머 우리들에게 필요한 답을 제공하는 자원이 풍부하다. 우리의 프로그래밍 질문에 주체적으로 답하는 습관은 대부분의 프로그래머들이 반드시 배워야 할 중요한 기술이다.

  - 파이썬이나 메인 언어가 출력하는 암호와 같은 에러 메시지 분석
  - 에러 메시지 문구를 검색 엔진에 그대로 입력하여 적합한 원인 찾기
  - 린터를 사용하여 개발자가 코드를 작성할 때의 오타와 잠재적인 버그를 짚게하기
  - 온라인 포럼에 글을 올리거나 누군가에게 이메일을 보내 문제를 해결하기

  마치 자신의 끊임없이 남들에게 답을 구하고 도움만 요청하는 것 같더라도, 낙심하지 말자.
  프로그래밍이라는 분야는 광대하며, 어느 누구도 그 모든 세부사항을 머리속에 전부 담을 수는 없다.
  주늑들지 말고 해결책을 능숙하게 찾는 데 주력한다면, 언젠가는 능숙한 파이썬 전문가의 반열에 오르게 될 것이다.
